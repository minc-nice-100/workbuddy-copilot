# WorkBuddy Copilot — 产品需求文档 (PRD)

> 版本：0.5.0-public-mvp
> 更新日期：2026-07-06
> 状态：公网 MVP 需求基线；改代码前必须先同步设计文档与测试方案

## 一、产品定位

WorkBuddy Copilot 是 PLC 学习场景下的实时学习观察与辅助系统。学员机通过 WorkBuddy hook 和桌面浮标把学习过程上报到公网中心服务；导师通过浏览器导师台观察 45 名学员的学习状态，并可向指定学员浮标发送文字提示或触发该学员上传完整对话内容。

本阶段目标是**公网可用的 MVP**，不再依赖局域网互通。隐私/合规的正式体系仍延后，但公网传输安全、角色 token、失败提示和数据隔离防线必须进入当前范围。

## 二、用户角色

| 角色 | 需求 | 呈现方式 |
|------|------|---------|
| 学员 | 不被 WorkBuddy hook 阻塞；通过浮标看到学习诊断、导师提示、同步状态；可主动向 Copilot 提问 | macOS 桌面浮标 |
| 导师 | 在公网导师台查看全部学员状态、会话内容和诊断；向指定学员发送提示；触发某学员上传完整对话 | 浏览器导师观察台 |
| 运维/管理员 | 部署公网服务、配置 token、排查同步/诊断失败、必要时删除学员数据 | 服务脚本 + 管理 API |

## 三、功能需求

### F1：公网中心服务连接

- 学员 hook、学员浮标、导师浏览器都连接同一个公网服务域名。
- 公网入口必须使用 HTTPS/WSS；本地开发可使用 HTTP。
- 应用服务保持单 uvicorn worker，避免内存态 WSRegistry/EventBus 分裂。
- 部署脚本和配置样例必须支持公网 host/base_url、student_id、student_token、mentor_token。

### F2：实时对话监控与学习分析

- WorkBuddy `UserPromptSubmit`：学员提问全文上报并入库。
- WorkBuddy `Stop`：hook 上传 transcript 尾部用于 LLM 分析，同时上传完整原文用于 raw transcript 留存。
- `/report` 必须快速返回 202；LLM 在后台执行，不阻塞 WorkBuddy。
- hook 保持 stdlib-only、fire-and-forget；任何异常都降级并返回 0。

### F3：导师观察台

- 三栏布局：学员列表 / 会话列表 / 时间线。
- 每个导师默认可查看全部学员。
- 时间线展示学员提问、AI 回复摘要、学习诊断、导师提示、全量上传后的逐轮对话内容。
- 配置 token 后，导师台必须能携带 mentor token 请求 REST 和 WS；未配置 token 只允许本地开发。
- 导师台当前不需要展示学员 WorkBuddy 的“正在查看哪个会话”。当前会话自动跟随只服务学员端体验。

### F4：导师向学员发送提示

- 导师可向指定学员浮标发送文字提示。
- 消息先入库，再按 `student_id` 定向发送；其他学员收不到。
- 学员浮标离线时，消息保留为未送达；重连后补拉并幂等渲染。
- 导师台显示发送中、已送达、失败状态。

### F5：导师触发学员上传完整对话

- 导师在某学员页面点击"同步该学员全部对话"。
- 服务端创建 `upload_requests` 并定向发送 `mentor_command` 给该学员浮标。
- 学员浮标在线时立即执行；离线时重连后补拉 pending request。
- 上传客户端只读取学员机本地 WorkBuddy 数据，只上传 `type == "message"` 行；工具输出、文件内容和 function result 不上传。
- 上传按会话逐个 POST，使用 sha 跳过未变会话。
- 同步状态必须可见：pending / running / done / failed。失败要给导师或学员明确提示，并允许重试。

### F6：历史会话诊断容错

- 上传完整对话后，服务端可后台对该会话跑 LLM 诊断。
- 不要求严格 exactly-once 或进程崩溃后自动完全恢复。
- 要求最小容错：LLM 失败或超时时不影响内容入库；状态标记为 failed 或待诊断；导师台能看到"内容已上传，诊断失败/可重试"。

### F7：学员主动向 Copilot 提问

- 学员可在浮标里主动输入问题。
- 服务端基于当前会话 raw transcript 或最近分析构造上下文，调用 LLM 回答。
- LLM 失败时返回固定降级回答，不阻塞 UI。
- 问答记录入库，便于后续回看。

### F8：学员端过程提醒

- 学员与 AI 对话过程中，如果出现低效模式，浮标可以轻提醒。
- 典型低效模式：反复无验证试错、上下文描述不清、让 AI 直接写但自己没有检查、方向跑偏、卡在同一个错误上。
- 提醒必须少而准，不能频繁打断；默认只在明确有帮助时出现。
- 过程提醒的提示词必须可自定义，便于持续迭代判断标准和语气。
- 当前版本由管理员/配置侧更新提示词即可；未来导师端可以更新“对所有学员生效”的全局提醒提示词，但本轮不做导师编辑 UI。

### F9：学员端浮标体验

- 浮标点击后打开一个轻量面板，不做复杂 tab。
- 顶部是“当前对话 / 最近对话切换”，用于切换查看哪个 WorkBuddy 会话的建议。
- 主体上半部分展示当前建议与导师提示；底部是“向技术助教提问”的输入框。
- 浮标客户端需要本地检测 WorkBuddy 当前会话，并默认自动切换到当前会话。
- 如果学员手动在面板里查看其他会话，面板打开期间不强行切走；面板关闭或重新打开时再自动跟随当前 WorkBuddy 会话。

### F10：数据隔离与删除

- `student_id` 是所有查询和写入的数据作用域键，但当前不是从认证身份派生的授权边界。
- `session_id` 仍按 WorkBuddy UUID 使用单主键，但公网环境下不能只依赖全局唯一假设；跨 student_id 的 session 冲突写入必须拒绝或忽略。
- timeline/transcript/reply 等读路径能带 `student_id` 的必须加作用域。
- 管理删除学员时，应级联删除该学员 reports、analyses、prompts、summaries、messages、raw_transcripts、mentor_messages、student_asks、upload_requests、sessions、students。
- 当前共享 `student_token` 只证明“学员端”角色；HTTP 请求体、查询参数和 `/ws` 的 `student_id` 由客户端提供。持 token 者目前可冒充其他学员，读取或确认其消息，或以其身份连接 WebSocket。
- 按 `student_id` 存储和查询不等于授权隔离。`auth.student_tokens` 与 `student_id_for_token()` 本期只是未接线的未来迁移接缝；只有路由从认证 principal 派生 `student_id` 并拒绝不匹配的客户端值后，才能缓解冒充风险。当前部署不得称为学员级或租户级数据隔离。

## 四、非功能需求

- **公网安全底线**：生产环境必须配置 token；公网流量必须 HTTPS/WSS。
- **角色隔离**：student token 不能访问导师 API；mentor token 不下发到学员机。
- **非阻塞**：hook、上传、LLM 失败不能影响 WorkBuddy 正常使用。
- **可观察性**：关键路径记录入口、分支、异常、返回值；导师可看到同步/诊断失败。
- **测试前置**：改代码前先更新测试方案；实现前先写能失败的测试。
- **服务端红线**：中心服务端不读取学员机本地 FS/DB；只有学员端脚本可以读取本机 WorkBuddy 数据并上传过滤后的内容。
- **可调提示词**：过程提醒的判断标准和表达方式不能硬编码死；应来自配置或数据库中的 prompt 版本。

## 五、当前不做

- 导师账号体系、per-student token 路由强制授权、审计后台（本期仅保留未接线 resolver 接缝）。
- Redis/MQ、多 worker、复杂重试队列、死信队列。
- 字段级脱敏、at-rest 加密、正式合规流程。
- Windows 浮标。
- 前端框架重构。

## 六、验收摘要

- 全量自动测试通过。
- 配置 token 后，导师台 REST/WS 可用；错误 token 被拒。
- 学员 token 无法访问导师 API。
- 浮标离线时导师触发同步不会静默丢失；重连后能补拉或显示失败。
- 跨学员 session_id 冲突不能污染他人会话。
- 公网部署脚本不再默认只绑定 `127.0.0.1`，并明确 HTTPS/WSS 前置。
- 学员端浮标能自动跟随当前 WorkBuddy 会话；过程提醒提示词可更新并影响后续提醒。
