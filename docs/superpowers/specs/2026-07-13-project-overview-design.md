# WorkBuddy Copilot 项目说明页设计

## 目标

制作一份面向内部同步与协作开发的单文件 HTML，让第一次接触项目的同事在不阅读源码的情况下，理解项目为什么存在、如何形成学习闭环、主要模块如何协作，以及当前从哪里可以参与。

## 叙事结构

1. 一句话说明项目价值：把学习过程变成可观察、可干预、可持续改进的闭环。
2. 解释动机：AI 学习过程隐藏、导师难以同时观察多人、卡点出现得早但反馈来得晚。
3. 用四步链路解释方法：旁路采集 → 中心分析 → 双端呈现 → 导师干预。
4. 用三栏架构图解释模块：学员端、中心服务、导师端。
5. 解释数据边界和消息可靠送达，保留足够技术深度但避免 API/代码细节。
6. 标注当前交付状态和新同事的参与入口，明确 Windows 实机验证仍是阻塞项。
7. 末尾提供术语小词典和项目文档入口，方便继续深入。

## 视觉方向

采用“浅色项目画布 + 荧光标记”的编辑型视觉：纸张色作为底，橙色代表关键行动，浅蓝代表中心服务，酸绿色代表学员端/导师端状态。通过更大的标题、清晰的编号节点、线性图标和三栏示意图建立信息层次；不依赖第三方前端库，项目内截图使用相对路径。

## 交互与兼容

- 顶部导航锚点跳转到主要章节。
- 当前章节使用滚动高亮，方便同步讲解时定位。
- 关键术语使用可展开的 `details` 组件。
- 04–06 的模块、数据边界和消息送达语义收进默认折叠的技术细节抽屉，按需展开。
- 04–06 改为三个可见的技术入口卡：每张先给一句白话摘要，点击后只展开对应主题的详细内容。
- 增加导师观察台真实截图和学员端 WorkBuddy + 浮标真实截图；两张图使用项目内相对资源路径，页面仍保留加载失败时的预览回退。
- 增加协作规模条形图和“学员端 → 中心服务 → 导师端”的事件/反馈路线图，减少对长段文字的依赖。
- 支持窄屏阅读和打印；尊重 `prefers-reduced-motion`。
- 说明内容基于 `README.md`、`docs/target-architecture.md`、`docs/prd.md` 和 `docs/technical-principles.md`，页面注明资料日期。
