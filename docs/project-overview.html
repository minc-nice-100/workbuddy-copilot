PLC learning infrastructure

把“学习过程”变成
看得见的闭环

WorkBuddy Copilot 是一个面向 PLC 学习场景的实时学习辅助系统：学员继续使用熟悉的 WorkBuddy，系统在旁边观察学习信号、识别卡点，并让学员和导师都能在合适的时机得到提醒。

它不替学员完成任务，也不改写 WorkBuddy 的对话流程。它做的是把原本藏在聊天记录里的学习状态，变成可以被及时看见、理解和回应的信息。

01 · Why

为什么需要它？

AI 辅助编程让“得到答案”变得很快，但学习的关键不只在答案。一个人可能是在理解后推进，也可能是在反复试错、复制粘贴，或者已经卡住却没有及时求助。

01 / 学习过程变隐形

只看最终产出，不知道中间发生了什么

同样一段代码、同样一个“完成”，背后可能对应完全不同的理解程度和学习状态。

02 / 导师观察有上限

45 位学员同时前进，靠人工很难全程跟上

导师需要一个全局视角，先看到值得关注的信号，再决定把时间花在哪里。

03 / 反馈总是来得太晚

卡点发生在当下，复盘往往已经是事后

越早发现低效对话、反复绕圈或理解断点，越有机会用很小的提醒避免大段时间浪费。

学习现场45位学员同时前进
观察与回应17–18位导师共享全局视图
统一事实1个中心服务承接所有状态

我们不替学员回答，而是帮助大家更早看见“正在发生什么”。

02 · How it works

一条低打扰的学习闭环

系统沿着 WorkBuddy 已经存在的对话生命周期工作。采集发生在旁路，分析集中在服务端，结果分别回到学员浮标和导师观察台。

01 / 捕捉信号

学员照常使用 WorkBuddy

Hook 在“学员发出问题”和“AI 回复完成”时收到通知，只带走必要的对话尾部，不阻塞原本的工作流。

hook → spool
02 / 理解上下文

中心服务把零散事件串起来

服务端保存完整的学习过程，调用 LLM 生成学习诊断、技术助教建议、AI 回答摘要和严重程度。

store + analysis
03 / 看见状态

学员与导师看到不同视角

学员浮标得到低频、即时的提示；导师观察台看到全体学员的状态、会话和时间线。

float + mentor desk
04 / 形成回应

导师可以定向发一条文字提示

导师消息只送给指定学员。在线实时到达，离线后也能补拉；只有学员真正处理并回执，才算送达。

observe → respond

03 · Architecture

三块拼起来，才是完整系统

可以把它理解成“学员端的信使 + 中心服务的大脑 + 导师端的观察台”。三者通过公网连接，但每一块都有清楚的责任边界。

04 · Modules

每个模块只做好一件事

协作开发时，可以先按这三个边界找到自己的位置。模块内部可以继续演进，但不要让“采集、分析、呈现、发送”互相穿透。

01 / Student

学员端：可靠地把信号送到

像一个不打扰工作的信使：先在本地稳稳接住，再通过网络送到中心。

  • Hook 保持 stdlib-only、fire-and-forget
  • 本地 spool 应对断网和进程重启
  • 共享核心让 macOS / Windows 复用同一套逻辑
02 / Center

中心服务：把事件变成判断

像一个学习信息中枢：保存事实、调用分析、统一分发，不依赖某一台学员机。

  • Service 负责编排业务流程
  • Store 负责唯一权威数据
  • EventBus 让新增推送渠道不必改业务核心
03 / Mentor

导师端：从全局观察到精准回应

像一块班级控制台：先判断哪里值得关注，再把一句有上下文的提示送到对应的人。

  • 观察全部学员，不预设固定负责关系
  • 实时看状态、会话和时间线
  • 消息按 student_id 定向，送达有回执
05 数据边界学员机采集暂存，中心服务解析并统一保存。 +
05 · Data boundary

数据从哪里来，应该去哪儿？

这是协作时最需要共同遵守的边界。它决定了系统能不能从“单机演示”走向多机使用，也决定了出了问题时我们能不能追溯。

学员机负责：采集 + 暂存

本地知道 WorkBuddy 的发生了什么，但不把本地路径和数据库访问交给服务器。

读取只读取被允许的 transcript 尾部字节
保存原子写入本地 spool，等待发送
失败时异常降级并返回 0，不阻塞 WorkBuddy

中心服务负责：解析 + 统一事实

所有学员的学习状态都在中心汇总，页面和分析只读取中心自己的数据。

接收通过 student token 上报事件
分析在服务端解析、入库并调用 LLM
追溯完整对话与分析结果可以对应到会话
唯一权威源：copilot.db
06 消息送达先保存、再推送、后回执，断线也能恢复。 +
06 · Delivery semantics

“看到消息”不等于“送达”

导师消息是这个项目里最容易被低估的一条链路。WebSocket 只能负责快，真正的送达要以学员端处理成功后的回执为准。

01 / 写入

服务端先保存消息

即使学员暂时离线，消息也不会因为“当下没连上”而消失。

02 / 推送

在线时走实时通道

在线的学员立刻看到；断线的学员之后通过补拉恢复。

03 / 处理

学员端先保存“待回执”

只有真正渲染和处理过的消息，才有资格进入下一步。

04 / 确认

服务端收到 ack 才标记送达

这个顺序保证重连、重启、响应丢失时仍然可以恢复。

at-least-once delivery · 至少一次投递 · 不把“发送成功”误当作“送达成功”

项目现在走到哪一步？

Open collaboration

新同事可以从哪里加入？

先理解边界，再认领一个完整的小闭环。下面这些方向都能独立验证，也能直接提升下一轮 rollout 的质量。

01Windows 实机验证：补齐 W0/W1 证据，推动 Windows 发布门从 blocked 走向可判断。
02导师观察台体验：优化状态、时间线、消息发送与错误提示，让导师更快做判断。
03分析与重试体验：围绕 pending / running / done / failed，把后台分析的不确定性讲清楚。
04测试与可观测性：沿着 P0/P1/P3 验证，确保“功能坏了时测试真的会红”。
08 · Glossary

几个词，先这样理解

第一次参与时，不需要一次读完所有代码。先把这些词和它们在系统里的位置对上号，再从对应模块往下钻。

Hook

WorkBuddy 在关键事件发生时调用的小脚本。它只负责把事件交给本地发送链路，不负责分析，也不应该拖慢 WorkBuddy。

Student Core

跨平台的学员端共享核心，负责本地暂存、网络发送、重连、去重和消息回执。它不依赖 macOS 或 Windows 的界面 API。

copilot.db

中心服务自己的 SQLite 数据库，是服务器唯一权威源。学员机上的 WorkBuddy 数据库不属于服务器的读写范围。

EventBus

服务内部的消息分发器。业务服务发布“分析完成”或“导师消息”等事件，浮标和导师台分别订阅自己需要的部分。

WebSocket / WSS

一种保持连接的实时通道。它让新事件可以主动推到客户端，但不单独承担可靠送达；可靠性由持久化和回执保证。

单 worker

当前服务必须以单个应用进程运行，因为 EventBus 和连接注册表在进程内。拆成多个 worker 会让广播和定向消息路由失去统一视图。
