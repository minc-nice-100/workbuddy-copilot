# 架构设计提问知识库

> 通用版。设计任何新系统/新模块时，按此清单提问，避免遗漏关键维度。
> 配套使用：`architecture-design-prompt.md`（架构设计助手提示词）

---

## 一、提问的价值：需求 → 设计 → 审视 → 精简

好的架构设计不是一次产出的，而是通过提问反复打磨出来的。典型的有效提问路径分三轮，每一轮都在质疑上一轮的结论：

### 第 1 轮：需求转架构

**问什么**：把需求说清楚，明确核心约束，尤其是**不对称需求**。

**价值**：
- 先锁定核心约束再设计，避免方案跑偏
- 不对称需求（如"某些数据原样、某些用摘要""某些实时、某些可离线"）直接决定数据模型
- 把"要什么"说清楚再问"怎么做"

**背后思维**：需求驱动设计。模糊需求产出模糊架构，精确约束产出精确方案。

### 第 2 轮：架构审视四维度

**问什么**：架构是否合理？功能是否解耦？未来扩展性？测试友好性？

**价值**：
- 跳出"能不能实现"，追问"实现得好不好"
- 逼出"按角色分模块 vs 按领域分模块"的根本问题
- 引出事件总线的解耦思路
- "测试友好性"这一维度大多数人会漏，但它直接决定重构勇气

**背后思维**：不轻易接受方案，从四个正交维度交叉验证。方案不是"产出的"，是"质疑出来的"。

### 第 3 轮：MVP 精简与过度设计判定

**问什么**：哪些过度设计？哪些必要？哪些现在做哪些以后做？

**价值**：
- 对抗"什么都要"的本能
- 逼出核心判断准则：**数据决策 vs 代码决策**

**背后思维**：YAGNI + 重构成本不对称性。知道"什么时候不该做"比"会做"更难。

### 提问路径的元价值

三轮提问形成 **设计 → 质疑设计 → 精简设计** 的闭环。每一轮都在质疑上一轮的结论，避免一次设计到位的盲目自信。方案是反复打磨出来的，不是一次产出的。

## 二、架构审视完整问题清单

### A. 基础四维度

#### A1. 模块合理性
- 模块是按"用户角色"分还是按"领域职责"分？
- 是否有上帝类（一个模块承担 5+ 项职责）？
- 依赖方向是否单向？是否有循环依赖？

#### A2. 解耦性
- 模块间是直接调用还是通过契约/事件？
- 修改一个模块会不会被迫改另一个模块？
- 生产者和消费者是否知道彼此？

#### A3. 扩展性
- 加一个新功能是"加代码"还是"改代码"？
- 有没有清晰的扩展点（订阅者、插件、策略）？
- 哪些功能是"架构转折点"（需要改变事件流方向）？

#### A4. 可测性
- 业务逻辑能不能脱离 IO/框架单独测？
- 依赖能不能 mock/注入？
- GUI/IO 层和逻辑层是否分离？

### B. MVP 阶段的维度取舍（关键）

不是所有维度在 MVP 阶段都要同等投入。按以下原则取舍：

#### B1. 降级与容错 ⚠（MVP 默认考虑，不过度设计）

**关键问题**：
- 每个外部依赖（LLM、数据库、消息队列、第三方 API、文件系统）挂了的基础行为是什么？
- 挂了会崩溃还是优雅降级？恢复后能自愈吗？

**MVP 够用线**：**挂了不崩 + 能查日志 = 够了**。重试队列、死信队列、断路器、熔断器、限流这些可以不做。

**判断准则**：每个外部依赖问一句"它挂了我怎么办"。基础要求是挂了不崩，恢复后自愈。这不是过度设计，是工程常识。过度设计的边界是"要求挂了零损失 + 自动恢复"才需要全套容错机制。

#### B2. 边界条件 ⚠（MVP 默认考虑，基本守卫即可）

**关键问题**：
- 空输入会怎样？
- 超长输入会怎样？
- 并发输入会怎样？
- 状态过渡态（如切换瞬间）会怎样？

**MVP 够用线**：空输入跳过、超长输入截断、并发按 key 隔离。不需要穷举所有极端值，但关键路径的守卫要有。

**判断准则**：列出"输入的极端值"和"状态的过渡态"，对关键路径逐一验证守卫。LLM/外部模型输入必须有 token 预算守卫，空数据必须跳过，并发必须有隔离 key。

#### B3. 数据生命周期（MVP 需要考虑）

**关键问题**：
- 数据保留多久？过期怎么清理？
- 数据可能含敏感信息（密码、密钥、个人隐私），怎么处理？
- 用户有"删除我的数据"的权利吗？
- 备份策略？数据损坏怎么办？

**MVP 够用线**：预留 cleanup 接口（即使不调用）、敏感数据处理开关（脱敏模式）、备份策略（单文件定期复制即可）。保留时长可暂不实施，但接口要留。

#### B4. 可观测性（MVP 需要考虑）

**关键问题**：
- 系统挂了怎么知道？
- 关键指标有哪些？
- 日志在哪？怎么排查问题？

**MVP 够用线**：
- `/health` 返回服务状态 + 依赖连通性
- 日志写文件，含关键 ID（用户 ID、会话 ID、事件类型、状态）
- 关键指标接口（累计数、失败数）

不需要 Prometheus/Grafana，但出问题要能查日志定位。

#### B5. 成本（MVP 通常不考虑，除非明确提）

**MVP 态度**：除非明确提到成本约束，否则不深入分析。LLM/API 单次成本可简单估算，但不作为设计约束。

**升级条件**：明确提"成本预算""月费上限""规模化成本"时才深入分析。

#### B6. 性能与容量（MVP 通常不考虑，除非特殊要求）

**MVP 态度**：单机、单进程在 MVP 阶段几乎总是够用。不考虑并发瓶颈、写锁冲突、连接数限制。

**升级条件**：明确提"毫秒级响应""万级并发""低延迟实时""高吞吐"等特殊性能要求时才考虑。

#### B7. 安全（MVP 局域网可简化，公网部署必做）

**MVP 态度**：局域网/MVP 阶段加简单认证即可。公网部署或多租户时才需要完整鉴权、加密、输入校验。

#### B8. 兼容性

**关键问题**：
- 表结构改了，旧数据怎么办？
- API 改了，旧客户端怎么办？

**MVP 够用线**：预留迁移机制、API 版本前缀。

#### B9. 成功标准

**关键问题**：
- 怎么知道这个系统"成功了"？
- 量化指标有哪些？

**MVP 态度**：靠主观反馈，规模化后接埋点。

### 维度取舍速查

| 维度 | MVP 态度 | 够用线 |
|------|---------|--------|
| 模块合理性 | 必做 | 按领域分，无上帝类 |
| 解耦性 | 必做 | 生产者消费者解耦 |
| 扩展性 | 必做 | 识别架构转折点 |
| 可测性 | 必做 | 业务逻辑脱离 IO |
| 降级容错 | 默认考虑，不过度设计 | 挂了不崩 + 能查日志 |
| 边界条件 | 默认考虑，基本守卫 | 空跳过/超长截断/并发隔离 |
| 数据生命周期 | 需要 | 预留接口 + 脱敏开关 + 备份 |
| 可观测性 | 需要 | /health + 日志 + 指标 |
| 成本 | 通常不考虑 | 除非明确提成本约束 |
| 性能容量 | 通常不考虑 | 除非特殊性能要求 |
| 安全 | 局域网简化 | 公网部署才必做 |

**工程常识 vs 过度设计的边界**：降级/边界条件属于工程常识，默认考虑是负责任的态度，不算过度设计。但"重试队列+死信+断路器+限流+熔断"全套就是过度设计。判断线：**挂了不崩 + 能查日志 = MVP 够了**。

## 三、核心方法论（可复用）

### 方法论 1：数据决策 vs 代码决策

**判断一个设计决策该不该现在做**：

```
这个决策改的是"已落盘的数据"吗？
├── 是 → 现在必须做对（表结构、外键、字段截断、事件分流落库）
│        因为数据写入后改结构要迁移历史，成本随数据量线性增长
└── 否 → 能拖就拖（接口抽象、分层、DTO、事件总线）
         代码重构是原子操作，成本恒定，等需求来了再加
```

### 方法论 2：过度设计判定三问

面对任何一个"为了未来"的设计，问三个问题：

1. **现在有第二个消费者吗？**（没有 → 接口抽象是 YAGNI）
2. **现在有第二个实现吗？**（没有 → 策略模式是 YAGNI）
3. **补做成本真的高吗？**（不高 → 现在不做）

### 方法论 3：扩展点 vs 架构转折点

- **扩展点**：加功能 = 加代码（加订阅者、加 Analyzer、加表）→ 不用提前设计
- **架构转折点**：加功能 = 改架构（改变事件流方向、改变一致性模型）→ 要提前预判

判断：如果新功能需要"反向事件流"或"跨进程协调"，它就是转折点，值得 MVP 时留接缝。

### 方法论 4：留接缝的技巧

不提前实现，但留好"一行替换"的接缝：

| 接缝类型 | 现在怎么做 | 未来怎么替换 |
|---------|-----------|-------------|
| 事件总线 | 广播调用集中到一个函数 | 改成 `bus.publish(event)` 一行 |
| Repository | 新表操作单独建函数，不混现有 | 提接口 + 换实现 |
| Analyzer/策略 | schema 字段分块注释 | 按区拆类 |
| DTO | JSON 直接用 | 加转换层 |

### 方法论 5：提问的三层递进

设计任何系统时，按这个顺序提问：

```
第 1 层：需求 → 核心约束是什么？哪些需求是不对称的？
第 2 层：审视 → 合理吗？解耦吗？能扩展吗？能测吗？
第 3 层：精简 → 哪些过度？哪些必要？数据决策还是代码决策？
第 4 层：补全 → 降级了吗？边界条件呢？可观测性？
```

### 方法论 6：MVP 阶段的维度取舍

不是所有维度在 MVP 阶段都要同等投入：

**MVP 必须考虑（8 项）**：模块合理性、解耦性、扩展性、可测性、降级容错（默认考虑不过度设计）、边界条件（基本守卫即可）、数据生命周期（预留接口）、可观测性（/health+日志+指标）。

**MVP 阶段不考虑（除非特殊）**：成本（除非明确提）、性能容量（除非"毫秒级/万级并发"等特殊要求）。

**升级触发条件**：
- 明确提到"规模化""多用户""生产环境"→ 性能/成本升级为必做
- 明确提到"公网部署""多租户"→ 安全升级为必做
- 明确提到"长期运行""合规"→ 数据生命周期深化

**工程常识 vs 过度设计的边界**：
- 降级/边界条件属于**工程常识**，默认考虑是负责任的态度，不算过度设计
- 但"重试队列+死信+断路器+限流+熔断"全套就是过度设计
- 判断线：**挂了不崩 + 能查日志 = MVP 够了**；要"挂了零损失 + 自动恢复"才需要全套

## 四、提问清单速查卡

> 每次设计新系统/新模块时，过一遍这个清单。

### 需求层
- [ ] 核心约束是什么？哪些需求是不对称的（如"原样 vs 摘要""实时 vs 离线"）？
- [ ] MVP 边界在哪？哪些是"只看不介入"式的约束？
- [ ] 成功标准是什么？怎么量化？

### 架构层
- [ ] 模块按"角色"分还是按"领域"分？有没有上帝类？
- [ ] 依赖方向单向吗？有循环依赖吗？
- [ ] 生产者和消费者解耦吗？有事件总线吗？

### 扩展层
- [ ] 加新功能是"加代码"还是"改代码"？
- [ ] 有哪些"架构转折点"？（反向事件流、跨进程协调）
- [ ] 未来功能哪些必要哪些可选？转折点功能优先级？

### 测试层
- [ ] 业务逻辑能脱离 IO 单测吗？
- [ ] 依赖能 mock/注入吗？
- [ ] GUI/IO 和逻辑分离了吗？

### 精简层
- [ ] 哪些设计是 YAGNI？（无第二消费者/第二实现/补做成本低）
- [ ] 哪些是数据决策必须现在做？哪些是代码决策能拖就拖？
- [ ] 留了"一行替换"的接缝吗？

### 降级层 ⚠（MVP 默认考虑，不过度设计）
- [ ] 每个外部依赖挂了的基础行为？
- [ ] 挂了不崩 + 能查日志 = MVP 够了（重试队列/断路器/熔断暂不做）

### 边界层 ⚠（MVP 默认考虑，基本守卫即可）
- [ ] 空输入跳过？超长输入截断？并发按 key 隔离？
- [ ] 不需要穷举，但关键路径守卫要有

### 数据生命周期层（MVP 需要考虑）
- [ ] 预留 cleanup 接口？敏感数据脱敏开关？备份策略？

### 可观测性层（MVP 需要考虑）
- [ ] /health 状态 + 依赖连通性？
- [ ] 日志写文件（含关键 ID）？关键指标接口？

### 成本层（MVP 通常不考虑，除非明确提）
- [ ] 仅当明确提成本约束时分析

### 性能容量层（MVP 通常不考虑，除非特殊性能要求）
- [ ] 仅当明确提"毫秒级""万级并发""低延迟"时考虑

### 安全层（MVP 局域网可简化，公网部署必做）
- [ ] 谁能访问？认证授权？输入校验？
- [ ] 敏感数据怎么处理？传输加密？

### 兼容层
- [ ] 表结构变了旧数据怎么办？API 改了旧客户端怎么办？
- [ ] 版本化策略？

## 五、配套提示词

本知识库配套一个可直接使用的架构设计助手提示词，见 `architecture-design-prompt.md`。

提示词内置了上述方法论，让 AI 自动遵守：
- 数据决策 vs 代码决策准则
- 过度设计三问
- MVP 维度取舍（8 项必做 + 2 项按需）
- "挂了不崩+能查日志"够用线
- 不对称需求识别
- 架构转折点预判
- 提问规则（不确定时一次最多问 3 个）
- 留接缝技巧
- 默认假设（不主动追问规模/技术选型）
